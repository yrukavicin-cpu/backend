# Lesson 13, 2026-04-22

## Тема
Методологія розроблення програмного продукту з використанням технологій ООП. Патерни Decorator, Observer, Visitor.

## Відповіді
1. **У чому рекурсивність декорування згідно патерну Декоратор?**  
   Рекурсивність декорування полягає в тому, що декоратор має той самий інтерфейс, що й об'єкт, який він обгортає. Тому один декоратор можна обгорнути іншим декоратором, а потім ще одним.

   Наприклад, базовий об'єкт `Text` можна послідовно обгорнути в `BoldDecorator`, потім в `ItalicDecorator`, і кожен шар буде додавати свою поведінку.

2. **Як реалізувати патерн декоратор на C++?**  
   У C++ для цього зазвичай створюють спільний абстрактний інтерфейс, конкретний компонент і базовий клас декоратора, який зберігає вказівник або розумний вказівник на компонент.

   ```cpp
   #include <iostream>
   #include <memory>
   #include <string>

   class Notifier {
   public:
       virtual ~Notifier() = default;
       virtual std::string send() const = 0;
   };

   class EmailNotifier : public Notifier {
   public:
       std::string send() const override {
           return "Email";
       }
   };

   class NotifierDecorator : public Notifier {
   protected:
       std::shared_ptr<Notifier> notifier;

   public:
       explicit NotifierDecorator(std::shared_ptr<Notifier> notifier)
           : notifier(notifier) {}
   };

   class SmsDecorator : public NotifierDecorator {
   public:
       using NotifierDecorator::NotifierDecorator;

       std::string send() const override {
           return notifier->send() + " + SMS";
       }
   };

   int main() {
       std::shared_ptr<Notifier> notifier = std::make_shared<EmailNotifier>();
       notifier = std::make_shared<SmsDecorator>(notifier);

       std::cout << notifier->send();
   }
   ```

3. **Які відмінності реалізації декораторів на Java від C++?**  
   У Java всі об'єкти створюються через посилання, а пам'яттю керує збирач сміття. Тому декоратор зазвичай просто зберігає посилання на інтерфейс або базовий клас.

   У C++ потрібно уважніше керувати часом життя об'єктів. Для цього часто використовують `std::unique_ptr` або `std::shared_ptr`, а в базовому класі обов'язково роблять віртуальний деструктор.

4. **Навіщо одній частині програми може бути потрібно спостерігати за іншою частиною програми? Наведіть приклади?**  
   Це потрібно, коли зміна стану одного об'єкта повинна автоматично повідомляти інші об'єкти. Так програма стає менш зв'язаною: об'єкт, за яким спостерігають, не повинен знати конкретну логіку всіх спостерігачів.

   Приклади:
   - кнопка повідомляє обробник про натискання;
   - модель даних повідомляє інтерфейс, що треба оновити таблицю;
   - датчик температури повідомляє екран і систему сигналізації;
   - біржовий сервіс повідомляє підписників про зміну ціни.

5. **Як патерн Observer забезпечує незалежність об'єкта спостереження від конкретних спостерігачів?**  
   Об'єкт спостереження працює не з конкретними класами, а зі спільним інтерфейсом спостерігача, наприклад `Observer`. Він зберігає список підписників і викликає в них метод оновлення.

   Завдяки цьому можна додавати нові класи спостерігачів без зміни класу об'єкта спостереження.

6. **Чи може клас конкретного спостерігача не залежати від класу конкретного об'єкта спостереження?**  
   Так, може. Для цього спостерігач повинен залежати від абстракції, а не від конкретного класу. Наприклад, метод `update()` може приймати прості дані або посилання на базовий інтерфейс `Subject`.

   Якщо спостерігачу потрібні тільки температура, ціна або текст повідомлення, йому не обов'язково знати повний клас об'єкта, який надіслав подію.

7. **Яку інформацію і кому можна передавати через параметр(и) метода Update в патерні Спостерігач?**  
   Через `Update` можна передавати інформацію про зміну стану об'єкта спостереження. Її отримують усі підписані спостерігачі або конкретний спостерігач, якщо реалізація підтримує вибіркові повідомлення.

   Наприклад, можна передати:
   - нове значення температури;
   - тип події;
   - старе і нове значення;
   - посилання на об'єкт-джерело;
   - структуру з детальними даними події.

8. **Яка основна ідея патерну Visitor?**  
   Основна ідея патерну `Visitor` — винести операції над об'єктами в окремий клас-відвідувач. Об'єкти структури приймають відвідувача методом `accept`, а відвідувач виконує потрібну дію для конкретного типу елемента.

   Це корисно, коли є стабільна ієрархія класів, але потрібно часто додавати нові операції над її об'єктами.

9. **Скільки методів має бути в інтерфейсі класів відвідувачів?**  
   Зазвичай в інтерфейсі відвідувача має бути окремий метод `visit` для кожного конкретного класу елемента, який може відвідуватися.

   Наприклад, якщо є класи `Circle`, `Rectangle` і `Triangle`, то інтерфейс відвідувача може мати методи `visit(Circle&)`, `visit(Rectangle&)`, `visit(Triangle&)`.

10. **Які наслідки матиме поява нового похідного конкретного класу елементів відповідно патерну Visitor?**  
    Якщо додати новий конкретний клас елемента, наприклад `Ellipse`, потрібно буде змінити інтерфейс відвідувача і додати новий метод `visit(Ellipse&)`. Після цього всі конкретні відвідувачі повинні реалізувати цей метод.

    Тому `Visitor` зручно використовувати тоді, коли набір класів елементів змінюється рідко, а нові операції над ними додаються частіше.
